# 시스템 설계 결정을 위한 트레이드오프 분석 (DECISIONS)

본 문서는 선착순 예약 시스템을 설계하며 직면한 주요 기술적 쟁점들과, 각 대안의 트레이드오프를 고려하여 최종 아키텍처를 결정한 과정을 기록합니다.

---

## 쟁점 1. 재고 정합성 및 공정성 보장
### 상황
* 평시 50TPS 환경이나, 프로모션이 시작되는 특정 시각(00시)에 한정 수량(10개)을 두고 500~1,000TPS의 트래픽이 1~5분간 폭증하는 상황입니다.
* 여러 쓰레드가 동시에 잔여 재고(remained_stock)를 수정할 때 발생하는 `Lost Update`를 막아 엄격한 정합성을 지켜야 하며, 모두에게 동등한 기회를 제공해야 합니다.

### 선택지
1. **DB 비관적 락:** 데이터 자체에 배타 락을 걸어 충돌을 원천 차단.
2. **DB 낙관적 락:** 버전(Version) 컬럼을 두어 충돌 시 애플리케이션 단에서 재시도.
3. **Redis 분산 락:** 별도의 Redis 인프라를 활용해 애플리케이션 외부에서 락을 획득하고 진입.

### 왜 그렇게 판단했는지
* **낙관적 락의 한계:** 충돌이 잦은 '선착순' 도메인 특성상, 실패한 요청들이 수백 번씩 재시도를 수행하게 되어 애플리케이션과 DB 자원을 심각하게 낭비합니다.
* **비관적 락의 한계:** 가장 확실하지만, 1,000개의 요청이 락을 얻기 위해 DB 커넥션을 점유한 채 대기하므로 커넥션 풀 고갈을 유발해 다른 일반 API(숙소 조회 등)까지 장애를 전파시킬 위험이 큽니다.
* **최종 선택 (Redis 분산 락):** 인프라가 추가되는 비용이 있지만, DB에 도달하기 전 Redis 메모리 단에서 스핀 락/Pub-Sub을 통해 트래픽을 가볍게 줄 세울 수 있습니다. 이는 DB 부하를 최소화하면서도 **'먼저 락을 요청한 순서대로(공정성)'** 처리할 수 있는 가장 적합한 구조라고 판단했습니다.
  * `Redisson` vs `Lettuce`
    * `Redisson`은 스핀 락과 Pub-Sub을 통한 대기열 관리 기능이 내장되어 있어 효율성과 안정성이 확보되지만, 별도의 의존성이 추가됩니다.
    * `Lettuce`는 Redis 의존성 내에 내장되어있지만, 분산 락 구현을 위해서는 별도의 로직을 직접 작성해야 하므로 개발 및 유지보수 부담이 증가합니다. 
    * 따라서, **Redisson**을 선택하여 개발 효율성과 안정성을 확보하였습니다.

## 쟁점 2. 고가용성 확보 전략
### 상황
* 과제 요구사항에 따라 인프라 증설(Scale-out)이 제한적인 환경입니다.
* **평시 트래픽:** 50 TPS (일 432만 건)
* **이벤트 트래픽:** 00시 오픈 직후 5분간 최대 1,000 TPS (5분간 약 30만 건, 평시 대비 20배 폭증)
* 만약 Redis 인프라가 다운되어 1,000 TPS가 모두 DB 비관적 락으로 우회할 경우, 스프링 부트 기본 설정(HikariCP 10개, 톰캣 쓰레드 200개) 하에서는 즉각적인 커넥션 풀 고갈과 쓰레드 풀 마비가 발생합니다. 이는 예약 기능뿐만 아니라 전체 시스템이 중단됩니다.

### 선택지
1. ~~**서버 타임아웃 자연 대기:** DB 응답을 끝까지 기다리다 타임아웃 발생.~~
2. **비동기 메시지 큐 도입:** 이벤트 기반 비동기 메세지 큐를 통해 부하를 분산.
3. **서킷 브레이커 도입:** 장애율 임계치 도달 시 DB 접근 원천 차단 (채택).

### 왜 그렇게 판단했는지
* 10,000 TPS 이상의 초대형 트래픽이 아니라면, 인프라 복잡도와 유지보수 비용을 급증시키는 Kafka 도입(선택지 2)은 현재 요구사항에서 오버엔지니어링이라고 판단했습니다. (쟁점 6에서 추가 설명)
* **최종 선택 (서킷 브레이커):** `Resilience4j`를 적용하여, 에러율이나 타임아웃 비율이 상승하면 즉시 회로를 열고(OPEN) **`503 Service Unavailable`** 예외를 던지도록 설계했습니다.
* **트레이드오프 결론:** 일부 예약 요청은 503 에러를 받고 튕겨 나가겠지만, 이 구조 덕분에 **DB 커넥션 고갈을 막고 시스템 전체가 마비되는 상황을 방어**할 수 있습니다. 즉, 일부 요청의 실패를 감수하더라도 서버 전체의 **고가용성**을 보장하는 차선책을 선택했습니다.

## 쟁점 3. 연속 결제 요청 멱등성 처리
### 상황
* 사용자가 결제 버튼을 빠르게 여러 번 누를 경우, 멱등성을 보장해주지 못하면 중복 결제가 발생할 수 있습니다.

### 선택지
1. **DB Unique Constraint:** 주문/결제 테이블에 멱등성 키를 유니크 컬럼으로 설정하여 방어.
2. **Redis 멱등성 키 기반 방어:** API 호출 진입점에서 즉시 검증 및 차단.

### 왜 그렇게 판단했는지
* DB 제약 조건을 활용하면 중복 요청이 결국 DB 커넥션을 타고 들어가 에러를 발생시키므로 불필요한 부하를 유발합니다.
* **최종 선택:** 클라이언트가 요청 시 발급한 고유 `idempotencyKey`를 조합하여, 진입점에서 `@Idempotent` 검증을 수행하도록 설계했습니다. DB에 도달하기 전 최전선에서 중복 요청을 튕겨내어 불필요한 리소스 낭비를 막았습니다.

## 쟁점 4. 결제 수단 확장성 및 복합 결제 구조 설계
### 상황
* 현재 신용카드, Y페이, 포인트를 지원하며 복합 결제(카드+포인트, Y페이+포인트)가 가능하지만, **카드와 Y페이는 혼용 불가**라는 비즈니스 규칙이 있습니다.
* 추후 새로운 결제 수단이 추가되어도 핵심 예약 비즈니스 로직(Booking API)의 수정이 없어야 합니다.

### 선택지
1. **Service 레이어 내 절차적 구현:** `if-else` 분기문으로 결제 수단을 검사하고 로직 실행.
2. **책임 분리를 통한 OCP(개방-폐쇄 원칙) 설계:** 결제 규칙 검증은 `Validator`로 분리하고, 도메인 내부에서 결제 정책을 캡슐화.

### 왜 그렇게 판단했는지
* 1번 방식은 결제 수단이 하나 추가될 때마다 핵심 로직인 `BookingService` 코드를 수정해야 하므로 변경에 매우 취약합니다.
* **최종 선택:**
  1. 비즈니스 제약 사항은 `BookingValidator`라는 별도의 컴포넌트로 책임을 분리하여 핵심 로직을 보호했습니다.
  2. 새로운 결제 시스템 `X페이` 도입 시, `PaymentMethod` Enum 클래스를 추가하고, `BookingValidator`에 새로운 검증 로직을 추가하는 방식으로 확장할 수 있도록 설계했습니다.
     (`BookService` 수정X)
  3. 결제 요청 정보를 특정 구현체에 의존하지 않도록 캡슐화하여 전달합니다. 이는 향후 서로 다른 결제 PG사로의 접근이 추가되거나 변경될 때, 인터페이스 기반의 캡슐화된 구조를 통해 로직의 변경 없이 새로운 PG 클라이언트 구현체만 추가하여 신속하게 확장하기 위함입니다.

## 쟁점 5-1. Redis 장애 시의 Fallback 전략
### 상황
* 쟁점 1에서 Redis 분산 락을 도입함으로써, **Redis가 죽으면 모든 예약 기능이 마비되는 단일 장애점**이 생성되었습니다.
* 결제/예약 도메인은 가용성이 핵심이므로, 인프라 장애가 서비스 시스템 전체가 중단되는 것을 방지해야합니다.

### 선택지
1. ~~**빠른 실패:** Redis 장애 감지 시 즉시 에러를 반환하고 클라이언트에게 재시도 안내.~~
2. **DB 비관적 락으로 즉시 우회:** 예외를 낚아채어 락 없이 진행하되, DB 레벨의 락(`FOR UPDATE`)을 사용하여 처리.

### 왜 그렇게 판단했는지
* 1번 선택지는 구현이 쉽지만, **인프라 장애로 인해 고객의 결제 기회를 박탈하게 되므로 비즈니스 측면에서 손해가 큽니다.**
* **최종 선택 (DB 비관적 락 우회)**: AOP 단에서 `Redis` 연결 끊김을 감지하면, 즉시 `try-catch`를 통해 비관적 락 쿼리(`findByIdForUpdate`)를 호출하는 Fallback 메서드로 트래픽을 우회시켰습니다.
* **트레이드오프:** 평상시에는 DB 부하를 줄이기 위해 Redis를 쓰지만, 비상 상황에서는 '성능저하'를 감수하더라도 **데이터 정합성 보장과 가용성 확보**를 최우선으로 두는 아키텍처를 선택했습니다.

## 쟁점 5-2. Fallback 동작 시 시스템 붕괴 방지 구조
### 상황
* Redis 장애로 인해 1,000TPS가 DB 비관적 락으로 Fallback하게 되면, 톰캣 쓰레드 풀마저 꽉 차 시스템 전면 붕괴가 예상됩니다.

### 선택지
1. **서버 타임아웃 대기:** 기본 설정대로 DB 응답을 기다리다 자연스럽게 타임아웃 발생.
2. **서킷 브레이커 (Circuit Breaker - Resilience4j) 도입:** 장애율이 임계치를 넘으면 DB로의 접근 자체를 차단.

### 왜 그렇게 판단했는지
* **최종 선택 (서킷 브레이커 도입):** DB가 감당할 수 없는 한계치를 넘었을 때 계속 요청을 보내는 것은 자멸하는 행위입니다. 예약 API에 `@CircuitBreaker`를 적용하여, 에러율이 상승하면 회로를 열고 즉시 `503 Service Unavailable` 예외를 뱉어내도록 설계했습니다.
* 이를 통해 극단적인 상황에서도 예약을 뺀 '숙소 정보 조회' 등 타 서비스는 정상 작동하도록 **장애의 반경을 격리**했습니다.

## 쟁점 6. 대규모 트래픽 처리를 위한 비동기 메시지 큐(Kafka) 도입 여부
### 상황
* 트래픽이 1,000 TPS를 넘어 10,000 TPS 규모로 성장할 경우, 동기식 처리 방식인 'Redis 분산 락 + DB 업데이트' 구조는 한계에 도달할 수 있습니다.
* 이에 따라 DB 부하를 근본적으로 차단할 수 있는 이벤트 기반 비동기 아키텍처 도입을 검토했습니다.

### 선택지
1. **현재 구조 유지 (동기식 처리):** Redis 분산 락을 통해 대기열을 순차적으로 제어하고, 클라이언트에게 즉각적인 응답(성공/재고 부족)을 반환.
2. **Kafka 도입 (비동기 처리):** 예약 요청을 일단 Kafka Topic에 쌓아두고(빠른 응답), 백그라운드의 Consumer가 큐에서 순서대로 꺼내어 DB 재고를 차감하는 방식.

### 왜 그렇게 판단했는지 (Trade-off 분석)
* **Kafka(비동기)의 장단점**
  * 큐가 트래픽을 받아 처리함으로 DB 중단을 막을 수 있지만, 예약을 시도한 사용자에게 즉각적인 성공/실패 여부를 알려줄 수 없는 **'최종 일관성'** 모델이 됩니다. 프론트엔드는 폴링이나 웹소켓으로 처리 결과를 따로 폴업해야 하므로 시스템 아키텍처와 UX 복잡도가 급증합니다.
  * 또한, Kafka는 트랜잭션 보장, 순서 보장, 장애 복구 등의 복잡한 운영 이슈가 수반되며, 단일 모듈보다는 MSA에 더욱 어울리는 인프라입니다.
* **최종 선택 및 결론:** 본 과제의 예상 트래픽(순간 최대 1,000 TPS) 수준에서는 인프라 복잡도를 높이는 Kafka 도입보다는, **Redis 분산 락으로 DB 부하를 제어하는 동기식 처리가 '시스템 복잡도 대비 투자 효율(ROI)'이 가장 높다**고 판단했습니다.
* 클라이언트에게 즉각적인 예약 확정 경험을 제공하는 것이 숙소 예약 도메인에서 중요하기 때문에 현재 구조를 채택하였으며, 향후 트래픽이 증가하거나 시스템의 복잡도가 높아지는 시점에는 Kafka 기반의 비동기 전환을 고려할 수 있도록 도메인 로직을 분리해 두었습니다.

## 쟁점 7. 엔티티 연관관계 매핑 (객체 참조 vs ID 간접 참조)
### 상황
* JPA를 사용 중이나, Order 엔티티에서 User와 Accommodation을 객체 참조(@ManyToOne)할지, ID(Long)로 간접 참조할지 고민했습니다.

### 선택 및 근거
* 각 도메인(회원, 숙소, 주문)의 강결합을 끊고 DDD의 애그리거트 루트 원칙을 지키기 위해 ID 간접 참조 방식을 채택했습니다. 
* 이를 통해 대용량 트래픽 상황에서 불필요한 지연 로딩(Lazy Loading) 및 N+1 쿼리를 원천 차단하여 성능을 최적화했으며, 향후 MSA로 분리될 때의 유연성까지 확보했습니다.
